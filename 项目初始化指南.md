# 项目初始化指南 — 用 AI 高效构建完整项目

核心思路：**三个文件驱动一切**，各司其职。

---

## 三文件定位

| 文件 | 读者 | 作用 |
|------|------|------|
| `创建需求.md` | AI + 开发者 | 告诉 AI **做什么**（功能规格） |
| `CLAUDE.md` | AI | 告诉 AI **怎么做**（行为准则） |
| `README.md` | 运维 + 接手者 | 告诉人**怎么用**（部署说明） |

三者不重复、不冲突：需求定义功能，CLAUDE 控制质量，README 指导部署。本指南说明前两个文件怎么写，以及整个项目流程怎么跑。

---

## 一、创建需求.md — 需求文档写法

不是普通的 PRD，是 **AI 能直接执行的规格书**。

### 1.1 技术栈精确到版本号和引用 URL

```markdown
## 技术栈
* 后端：.NET 8 + ASP.NET Core Razor Pages
* 前端：Bootstrap 3（CDN 远端引用）
* CDN URL：https://apps.bdimg.com/libs/bootstrap/3.3.4/css/bootstrap.min.css
* 部署：Windows Server + IIS
```

❌ 不写"使用前端框架" → ✅ 写具体 URL 和版本号，零歧义。

### 1.2 数据结构定义到边界条件

```markdown
## 列结构
1. 条件列：判定分组的依据列，列数不固定
2. 目标列：校验的对象列，列名格式统一（如 role、role1...role20）
3. 导出列：新增标识列（如"是否存在异常"，取值 是/否）
```

每个字段写明：最大数量、空值含义、编码格式。

### 1.3 规则用"统计口径 + 判定条件"格式

```markdown
## 规则1：名称
* 统计口径：具体怎么算（如"非空单元格个数"）
* 判定条件：什么情况触发（如"同一组内取值不统一 → 异常"）
```

❌ 不写"检查数据合法性" → ✅ 写"统计 X 列非空个数，组内比较，不一致则标记异常"。

### 1.4 输出格式精确到列名和顺序

```markdown
## 导出格式
* 列顺序：新增列1（首列） + 原始所有列
* 异常行取值：是 / 正常行取值：否
* 约束：原始内容完整保留
```

### 1.5 通用约束兜底

```markdown
## 技术约束
1. 自动识别列结构，无需手动配置
2. 动态适配列数变化
3. 特殊字符处理规则
4. 编码格式、部署环境要求
```

## 二、CLAUDE.md — AI 行为准则

完整内容见项目根目录 [CLAUDE.md](CLAUDE.md)，四条核心规则摘要：

| 规则 | 核心思想 | 实际效果 |
|------|---------|---------|
| **Think Before Coding** | 先想再写，不确定就提问 | 不会默默选错方案 |
| **Simplicity First** | 简单优先，不过度设计 | 没有不需要的抽象层 |
| **Surgical Changes** | 精准修改，不夹带私货 | diff 干净、审查省力 |
| **Goal-Driven Execution** | 目标驱动，每步可验证 | 改动自带验证，不会没测 |

**关键**：CLAUDE.md 控制 AI 的**输出质量**而非功能需求。没有它时 AI 倾向于过度设计、猜测需求、顺手重构。

## 三、README.md — 部署说明模板

新项目 README 应包含以下章节：

```markdown
# 项目名 — 一句话描述

## 技术栈          ← 表格：后端/前端/部署/调试
## 功能            ← 功能列表
## 项目结构         ← 目录树
## 环境要求         ← 开发 + 部署
## 本地开发调试     ← clone → run → 浏览器
## 部署             ← IIS/服务器配置步骤
## 输入格式要求     ← 文件编码、列结构、示例
## 核心规则         ← 规则表格
## 配置文件         ← appsettings 说明
## 常见问题         ← FAQ
```

## 四、开发流程模板

按以下顺序推进，每个阶段结束立即验证：

| 阶段 | 输入 | 产出 | 验证方式 |
|------|------|------|----------|
| 项目骨架 | 技术栈要求 | 项目结构 + 依赖 | `dotnet build` |
| 核心引擎 | 规则章节 | 纯逻辑类 | 控制台测试 + 真实数据 |
| 页面交互 | UI + 导出要求 | 前端页面 + API | 浏览器操作 |
| 踩坑修复 | 用户反馈 | 针对性修复 | 复现 → 修复 → 验证 |
| 部署上线 | 部署要求 | 发布包 + 配置 | 目标环境访问 |

## 五、关键原则

### 5.1 需求文档决定代码质量

模糊描述 → AI 猜测 → 猜错 → 重写。精确描述 → AI 执行 → 一次通过。

### 5.2 CLAUDE.md 是质量阀门

没有它时 AI 过度设计、猜测、重构。有了规则后可控可预测。

### 5.3 每个阶段要有验证

编译期 `build` → 逻辑期独立测试 → 交互期浏览器操作 → 部署期目标环境访问。

### 5.4 真实数据驱动调试

一个真实输入文件比文字描述更有用，立即暴露边界条件。

### 5.5 发布目录防嵌套

`dotnet publish -o release` 输出在项目内部时，下次发布会把旧输出也打包，产生 `release\release\...` 嵌套。

**修复**：在 `.csproj` 中排除：
```xml
<DefaultItemExcludes>$(DefaultItemExcludes);release\**</DefaultItemExcludes>
```

### 5.6 每次变更后同步 Git

```bash
git add -A
git status
git commit -m "类型: 简述"
git push
```

提交格式如 `v3: 授权码登录 + 比对页面`，让后续接手者快速了解每个版本。

### 5.7 子路径部署注意路径

部署到 `/MatricesCheck` 等子路径时，使用 Razor Helper 而非硬编码路径：

```csharp
// ❌ 部署到子目录会炸
<img src="/logo.png" />
<a href="/Index?handler=Download">

// ✅ 自动适配
<img src="@Url.Content("~/logo.png")" />
<a href="@Url.Page("/Index", "Download")">
```

### 5.8 POST 后重定向防重复提交

`return Page()` 会导致浏览器刷新时弹出"确认重新提交表单"。改为 **PRG 模式**：

```csharp
// ❌ POST 后直接返回页面
return Page();

// ✅ 重定向到 GET，状态通过 TempData 传递
return RedirectToPage();
```

GET 方法从 TempData 恢复状态渲染，刷新只是重新 GET。

## 六、需求文档模板

```markdown
## 技术栈
* 后端框架 + 精确版本号
* 前端 UI + 引用方式（具体 URL）
* 部署环境

## 数据结构
* 每种列/字段的定义和边界条件
* 输入格式
* 输出格式（精确到列名和顺序）

## 核心规则
* 规则1：统计口径 + 判定条件
* 规则2：统计口径 + 判定条件
* ...

## 页面展示要求
* UI 框架 + 样式来源
* 异常标记方式
* 交互行为

## 导出要求
* 文件格式 + 编码
* 列顺序 + 字段取值规则

## 通用技术约束
* 兼容性要求
* 边界条件
* 特殊字符处理
```

## 七、适用场景

**适合**：数据校验/报表工具、CRUD 管理系统、有明确输入输出规则的项目、单人/小团队快速出活。

**不适合**：探索性项目（需求本身不确定）、重度交互前端应用、需要大量架构决策的系统。
