# 项目初始化指南 — 用 AI 高效构建完整项目

核心思路：**两个文件驱动一切**。

---

## 文件结构

```
项目目录/
├── 创建需求.md        ← 需求文档（给人看，也给 AI 看）
├── CLAUDE.md          ← AI 行为准则（控制 AI 输出质量）
└── README.md          ← 部署说明（给运维/接手者看）
```

---

## 一、创建需求.md — 需求文档写法

这不是普通的 PRD，是 **AI 能直接执行的规格书**。

### 1.1 技术栈精确到版本号和引用 URL

```markdown
## 技术栈
* 后端：.NET 8 + ASP.NET Core Razor Pages
* 前端：Bootstrap 3（CDN 远端引用）
* CDN URL：https://apps.bdimg.com/libs/bootstrap/3.3.4/css/bootstrap.min.css
* 部署：Windows Server + IIS
```

❌ 不写"使用前端框架" → ✅ 写具体 URL 和版本号，零歧义。

### 1.2 数据结构定义到边界条件

```markdown
## 列结构
1. 条件列：判定分组的依据列，列数不固定
2. 目标列：校验的对象列，列名格式统一（如 role、role1...role20）
3. 导出列：新增标识列（如"是否存在异常"，取值 是/否）
```

每个字段写明：最大数量、空值含义、编码格式、是否可为空。

### 1.3 规则用"统计口径 + 判定条件"格式

```markdown
## 规则1：名称
* 统计口径：具体怎么算（如"非空单元格个数"）
* 判定条件：什么情况触发（如"同一组内取值不统一 → 异常"）
```

❌ 不写"检查数据合法性" → ✅ 写"统计 X 列非空个数，组内比较，不一致则标记异常"。

每条规则独立，AI 逐条实现 + 逐条验证。

### 1.4 输出格式精确到列名和顺序

```markdown
## 导出格式
* 列顺序：新增列1（首列） + 原始所有列
* 异常行取值：是
* 正常行取值：否
* 约束：原始内容完整保留，不做修改
```

### 1.5 通用约束兜底

```markdown
## 技术约束
1. 自动识别列结构，无需手动配置
2. 动态适配列数变化
3. 特殊字符处理规则（如 * 作为普通文本不启用通配）
4. 编码格式、部署环境要求
```

## 二、CLAUDE.md — AI 行为准则

控制 AI 的**输出质量**而非功能需求，四条核心规则：

### 2.1 Think Before Coding — 先想再写

```markdown
Before implementing:
- State your assumptions explicitly. If uncertain, ask.
- If multiple interpretations exist, present them — don't pick silently.
- If something is unclear, stop. Name what's confusing. Ask.
```

**效果**：AI 遇到模糊点会先提问确认，不会默默选错方案。

### 2.2 Simplicity First — 简单优先

```markdown
- No features beyond what was asked.
- No abstractions for single-use code.
- If you write 200 lines and it could be 50, rewrite it.
```

**效果**：不会出现策略模式、工厂模式等不需要的抽象层。

### 2.3 Surgical Changes — 精准修改

```markdown
- Don't "improve" adjacent code, comments, or formatting.
- Match existing style, even if you'd do it differently.
- Every changed line should trace directly to the user's request.
```

**效果**：diff 干净不夹带私货，审查省力。

### 2.4 Goal-Driven Execution — 目标驱动

```markdown
Transform tasks into verifiable goals:
- "Add validation" → "Write tests for invalid inputs, then make them pass"
- "Fix the bug" → "Write a test that reproduces it, then make it pass"
```

**效果**：每次改动自带验证，不会"看起来好了但其实没测"。

## 三、开发流程模板

按以下顺序推进，每个阶段结束立即验证：

| 阶段 | 输入 | 产出 | 验证方式 |
|------|------|------|----------|
| 项目骨架 | 技术栈要求 | 项目结构 + 依赖 | `dotnet build` |
| 核心引擎 | 规则章节 | 纯逻辑类（无外部依赖） | 控制台测试项目 + 真实数据 |
| 页面交互 | UI + 导出要求 | 前端页面 + API | 浏览器操作真实数据 |
| 踩坑修复 | 用户反馈 | 针对性修复 | 复现 → 修复 → 验证 |
| 部署上线 | 部署要求 | 发布包 + 配置 | 目标环境实际访问 |

## 四、关键原则

### 4.1 需求文档决定代码质量

模糊描述 → AI 猜测 → 猜错 → 重写。
精确描述 → AI 执行 → 一次通过。

### 4.2 CLAUDE.md 是质量阀门

没有它时 AI 倾向于过度设计、猜测需求、顺手重构。有了规则后行为可控可预测。

### 4.3 每个阶段要有验证

不要"写完再说"：
- 编译期：`build`
- 逻辑期：独立测试项目
- 交互期：浏览器真实操作
- 部署期：目标环境实际访问

### 4.4 真实数据驱动调试

一个真实的输入文件比任何文字描述都有用，能立即暴露边界条件和逻辑漏洞。

### 4.5 发布目录防嵌套

`dotnet publish -o release` 如果输出目录在项目内部，下次发布会把旧输出也当作项目文件打包，产生 `release\release\release\...` 无限嵌套。

**修复**：在 `.csproj` 的 `<PropertyGroup>` 中排除发布目录：

```xml
<DefaultItemExcludes>$(DefaultItemExcludes);release\**</DefaultItemExcludes>
```

**清理**：如果已经嵌套，直接删除整个 `release\` 目录后重新发布。

## 五、需求文档模板

```markdown
## 技术栈
* 后端框架 + 精确版本号
* 前端 UI + 引用方式（具体 URL）
* 部署环境

## 数据结构
* 每种列/字段的定义和边界条件
* 输入格式
* 输出格式（精确到列名和顺序）

## 核心规则
* 规则1：统计口径 + 判定条件
* 规则2：统计口径 + 判定条件
* ...

## 页面展示要求
* UI 框架 + 样式来源
* 异常标记方式
* 交互行为

## 导出要求
* 文件格式 + 编码
* 列顺序 + 字段取值规则

## 通用技术约束
* 兼容性要求
* 边界条件
* 特殊字符处理
```

## 六、适用场景

**适合**：数据校验/报表工具、CRUD 管理系统、有明确输入输出规则的项目、单人/小团队快速出活。

**不适合**：探索性项目（需求本身不确定）、重度交互前端应用、需要大量架构决策的系统。
